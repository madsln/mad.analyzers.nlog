## Architektur-Entscheidung

### ADR-006: Kein CLI-Framework

**Problem:** Soll die CLI-Schicht ein externes Framework (`System.CommandLine` o. ä.)
verwenden oder manuell implementiert werden?

**Entscheidung:** Manuelles Argument-Parsing ohne externe Abhängigkeit.

**Begründung:** Drei Subkommandos, vollständig bekanntes Argument-Schema, zero-dependency-
Policy der restlichen Codebasis, geringer Implementierungsaufwand (< 150 Zeilen).

→ Vollständig dokumentiert in [ADR-006](../adr/ADR-006-CLI-No-Framework.md).

## Implementierungsdetails

### `CliRouter`

- `args[0]` bestimmt das Subkommando via `switch`-Expression.
- `--help`, `help`, `-h` und leere Argumente werden alle zur globalen Hilfe geleitet.
  Bei `args.Length == 0` wird Exit-Code 1 zurückgegeben (kein Subkommando angegeben),
  bei explizitem `--help` Exit-Code 0.
- Unbekannte Subkommandos: Fehlermeldung auf `stderr` + Hilfe + Exit-Code 1.

### `DemoCommand`

- Parst `--out` mit linearer Suche (identisches `ParseArg`-Hilfsmuster in allen Commands).
- Gibt Erfolgs-Zusammenfassung mit absolutem Pfad aus (via `Path.GetFullPath`).

### `TestCommand`

- Prüft Existenz von `--source` (Exit 2) und ob `.cs`-Dateien vorhanden sind (Exit 2).
- Standard-`--out` bleibt wie bisher: `../../test/mad.analyzers.nlog.common.Test/generated`
  relativ zu `--source`.

### `AllCommand`

- Delegiert an `DemoCommand.Run` und `TestCommand.Run` mit synthetisch zusammengesetzten
  `args`-Arrays – kein Code-Duplikat.
- Bricht ab, wenn `DemoCommand` einen Fehler zurückgibt (Exit-Code-Weiterleitung).

### `HelpTexts`

- Alle Hilfetexte als Raw-String-Literale (`""" ... """`), klar lesbar.
- Zentral in `Cli/HelpTexts.cs`, von allen Commands und dem Router aufgerufen.

### `Program.cs`

Genau gemäß Workload-Spezifikation:

```csharp
try
{
    return CliRouter.Run(args);
}
catch (Exception ex)
{
    Console.Error.WriteLine($"Fehler: {ex.Message}");
    return 3;
}
```
